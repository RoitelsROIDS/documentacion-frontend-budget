Instalación y primeros pasos

1. Framework, Lenguaje y Biblioteca de Componentes de UI​

La aplicación utiliza el framework/librería de React Js y la biblioteca de componentes Material-UI para construir su interfaz de usuario. Material-UI proporciona una amplia gama de componentes predefinidos y estilizados que siguen los principios de diseño de Material Design. En cuanto al lenguaje de programación utilizado en el proyecto, se desarrolla en TypeScript, que es un superconjunto tipado de JavaScript. TypeScript agrega características de tipado estático a JavaScript, lo que permite detectar y corregir errores en tiempo de compilación.

¿Por qué Typescript?​

Algunos de los beneficios de TypeScript son los siguientes:

  • Tipado estático: TypeScript permite declarar tipos de variables, parámetros de función y valores de retorno, lo que proporciona mayor seguridad y detecta posibles errores durante el desarrollo. Esto ayuda a evitar errores comunes y mejora la calidad del código.

  • Mejor soporte para IDEs: Los IDEs (Entornos de Desarrollo Integrados) ofrecen un mejor soporte para TypeScript en comparación con JavaScript. Proporcionan funciones avanzadas como autocompletado, navegación de código, refactoring y verificación de tipos en tiempo real, lo que agiliza el desarrollo y reduce errores.

  • Mayor mantenibilidad: Al utilizar tipado estático, el código escrito en TypeScript es más legible y autodocumentado. Esto facilita el mantenimiento y la colaboración en proyectos a largo plazo, ya que es más fácil comprender el propósito y la estructura del código.

  • Compatibilidad con JavaScript: TypeScript es compatible con JavaScript existente, lo que significa que se puede usar en proyectos existentes sin necesidad de realizar cambios drásticos. Los archivos JavaScript se pueden incluir en un proyecto de TypeScript, y TypeScript puede consumir y emitir código JavaScript.

  • Mejora la productividad: TypeScript permite utilizar características modernas de JavaScript antes de que sean oficialmente admitidas en todos los navegadores. Además, proporciona herramientas y características adicionales, como módulos, clases y interfaces, que facilitan el desarrollo de aplicaciones escalables y mantenibles.

2. Requisitos previos​

Para clonar e instalar el proyecto, necesitarás tener instalado en tu sistema lo siguiente:

  • Git: Git es un sistema de control de versiones distribuido ampliamente utilizado. Asegúrate de tener Git instalado en tu sistema antes de clonar el repositorio. Puedes obtener Git desde el sitio web oficial de Git: https://git-scm.com/

  • Node.js y npm: Node.js es un entorno de ejecución de JavaScript del lado del servidor, y npm (Node Package Manager) es el gestor de paquetes de Node.js. Asegúrate de tener instalada una versión reciente de Node.js, que también incluirá npm. Puedes descargar Node.js desde el sitio web oficial: https://nodejs.org/

  • Yarn: Como gestor de paquetes alternativo a npm, Yarn es una herramienta de gestión de dependencias que ofrece algunas ventajas adicionales, como una mayor velocidad de instalación y una mejor gestión de la caché.Puedes instalar Yarn siguiendo las instrucciones disponibles en su sitio web oficial: https://yarnpkg.com/

3. Clonación del repositorio e instalación de las dependencias​

Para clonar el repositorio y configurar el entorno de desarrollo local, sigue los siguientes pasos:

a. Abrir una terminal y ejecutar el siguiente contacto para clonar el repositorio

git clone https://github.com/roitels/budgets-react-frontend.git

b. Navega al directorio del proyecto

cd /budget-react-frontend

c. Instala las dependencias necesarias ejecutando el comando npm/yarn install. Este comando instalará todas las dependencias listadas en el archivo package.json.

npm install
yarn install

d. Ejecutar el comando para iniciar el repositorio en el localhost:3000

npm run start
yarn start

4. Colaboración​

Como método de trabajo para los diferentes integrantes del equipo de desarrollo front-end se recomienda lo siguiente:

  • Una vez clonado el repositorio, trabajar a partir de ramas/branchs. Por cuestiones organizativas (para reconocer quien es el propietario de la rama y en que se encuentra trabajando) se recomienda nombrar las ramas/branches de la siguiente manera:
// nombre del integrante / acción que se lleva a cabo (feat: si es para el desarrollo de una nueva pantalla, elemento, componente o feature | fix: si es para continuar el desarrollo o realizar modificaciones) | pantalla o componente en el cual se esta trabajando

Ej: lucio/feat/login-screen (esta rama dice el nombre del integrante, y que se encuentra desarrollando la pantalla de inicio de sesión)

Ej: lucio/fix/budget-charts (esta rama dice el nombre del integrante, y que se encuentra arreglando el componente que renderiza los gráficos en la pantalla Budget)

  • El repositorio donde se encuentra alojado el código posee 2 ramas principales: dev (rama para el desarrollo) y main (rama para producción)
Importante!

Una vez finalizado el trabajo en la rama personal, la forma correcta de proseguir es publicar la rama en el repositorio y luego realizar desde github un pull request hacia la rama DEV, NUNCA debe realizarse un merge de la rama personal directamente a MAIN. Primero se realizar el pull request a la rama DEV para corroborar que el nuevo código agregado no entre en conflicto con las ramas publicadas, la función de la rama DEV es corroborar que todo el código funcione y este en orden antes de llevarlo a producción. Si el código mergeado a DEV no produce ningún conflicto, se procede a realizar una nueva pull request de DEV hacia MAIN para llevar todo el nuevo código a producción.
